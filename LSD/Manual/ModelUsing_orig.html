Using a Lsd Model

Introduction

The best way to learn to use a Lsd model is, like any software, to try and err. In the following is presented a brief tutorial listing the main steps in using a Lsd model, as guidelines to start using Lsd models.After this tutorial you should be able to fully exploit an existing model, and to explore the Lsd interfaces even in the parts not covered here. Refer to the main Lsd manual for a complete description of the Lsd interfaces.

To run a simulation exercise with Lsd it is used:

  • a Lsd model program. This is a program mixing the C++ code implementing equations for the model and the general Lsd code to deal with the model. The modeller has created the code for the equations, and users cannot modify it by running the model program.
  • a model configuration. This is the definition of the model prepared by the modeller. It contains the description in terms of entities (Objects), Variables and Parameters. The configuration contains also all the necessary information used to run a simulation: the number of Objects, the initial values for Parameters, the number of time steps, etc. Users can easily modify each of these values to test different configurations.
Using Lsd entails the access to several windows, and sometime the one to use is hidden behind other ones.
 
Step  Operation Instr.  Menu choices and Comments
1 Run the Lsd model program LMM
  1. Model/Select
  2. Model/Run
Choose the model you want to explore, and then run the corresponding Lsd model program. LMM automatically starts the Lsd model program for you (ensuring that it has been properly compiled). The model interfaces are initially empty, since no configuration is loaded.
2 Load a configuration file Lsd
  • File/Load
A model configuration contains the model description, the initial data, and everything it is required to run a simulation. The modeller has prepared at least one configuration, while more will be created when using the model.
3 Give a first look to the model Lsd 
  • See the structure of the model
  • Help/Model Help
  • Help/Model Report
Although you may already run a simulation, spend some time to look at how Lsd represents a model.
The "scheleton" of the model are Objects, elements containing Parameters, Variables, or other Objects. Therefore, a model is represented by a hierarchy of Objects, whose "top" element is conventionally called Root. Root is the only Object that is always present in a model with a single instance, while the other Objects can be multiplied in as many instances as desired. 

It is possible to explore the model with the Browser (square window on the top left part of the screen), which shows one Object's content. This is shown with some headers and two list boxes: Variables (and Parameters) and the Objects contained, if any. Variables are indicated with a integer number (the lagged values used in the model for that Variable), and Parameters are indicated with the letter "P". Variables labels can also be double-clicked to set their options (see below).
The Browser can be moved from one Object to another double-clicking on the descendants of an Object (to go "down") or on the name of the Object's "parent" (to go "up").
When a model is loaded a graphical representation of the model is shown (on the right of the Browser). It is also possible to click on the symbol of an Object in the graphical representation of the model to move there with the Browser.
The model help contains the verbal descriptions of the elements, and the model report the actual code used for the equations and other relevant information on the model structure. The two documents contain links to each other bringing at your attention all the relevant information for each element of the model.
4 Run a simulation Lsd
  • Run/Run
  • Observe the Run Time Plot
Run a simulation exercise accepting all the default initialization defined by the modeller. Before running the symulation, the system issues a confirmation message listing the options used for the simulation.
A simulation run consist in the Lsd model manager that explores every Object in the model computing the equation for their Variables. The Lsd model manager ensures that the values Variables used in the equation of other Variables are updated as required, independently from the actual exploration of the model.
During the simulation run some Variables are plotted in a self-scaling window. More series are saved in the memory for later analysis. During a simulation it is possible to use the buttons in the Log window:
  • Stop: abort the simulation
  • Fast: make run the simulation faster not issuing any message, if possible (minimize the Run Time Plot window for accellerating sensibly the simulation)
  • Observe: run the simulation in normal (message issuing) mode, if requested by the configuration.
  • Debug: interrupt the simulation when an equation marked to be debugged is computed. The simulation run may be resumed after that.
5 Analyse the results Lsd
  • Data/Analysis of Results
Create graphical representations of the data produced in the simulation. This module can be used with the data stored in memory after a simulation run, or using the data previously saved on a file (choose Simulation for using the data from the latest exercise).
The overall philosophy of this module is to list the elements available for analysis on the left box. Some of these should be selected and moved to the central box for elaboration, activated with the button in the lower part of the window. The right box contains the list of the graphs produced.
Frequently the large number of graphs produced auses a mess to find one specific graph, or even the main control window. To avoid this, click on one of the items in the right box to pop the corresponding graph up in the foreground. Double-clicking any graph brings the main control window in the foreground.
See here for the manual on this topic.
6 Interpret the results Netsc.
  • use Model Help and Model Report
Search in the documentation the elements of interest and look at their functioning in the model to understand the motivation for ther results obtained.
Notice that the equations' code in the Model Report, although obscure to non-Lsd programmers, contains extensive comments (in green text), translating the overall meaning of the code.
7 Reload the configuration Lsd
  • File/Reload
Reset the model to contain again the values for the beginning of the simulation.
After a simulation exercise Lsd contains in memory the values computed at the last time step. The system prevents the use of these values as the beginning of a new simulation run, and therefore a new configuration must be loaded.
Note that it is possible to save the model status as new configuration, which can be loaded to continue a simulation exercise.
8 Edit the number of instances for the Objects Lsd
  • Data/Set Number of Objects/All types of Objects
Accept or modify the number of Objects. See the Lsd help on this
9 Edit the initial values Lsd
  • Browse the model
  • Data/Init. Values
Change the initial values used to start a simulation. They concern the Parameters and the Variables used with lagged values. These changes are made on one single Object type per time, and it is necessary to insert one value for each Object instance. See the Lsd manual on this. 
10 Edit the simulation settings Lsd
  • Run/Sim. Settings
Choose the number of time step per simulation, the seed for the random generator, and number of simulation exercises to be run in a sequence (each with the same initialization values and different random generator seed). See the Lsd manual on this.
Note that running more than a simulation exercise cause the system to save the data in files.
11 Edit the choices on which
series to save, to plot
at run time, or to debug		 Lsd 
  • Browse the model
  • double-click on the Variable of interest
For each Variables and Parameters it is possible to decide whether:
  • it must be included in the data saved for analysis after a simulation (To Save)
  • it must appear in the Run Time Plot, to observe its values during the simulation run (Run Time Plot)
  • it should cause the simulation to be interrupted immediately after its equation has been computed (Debug). Only for Variables
See the Lsd manual on this.
These choices can be changed individually for each element, or can be set  off for the whole model using Remove ... in menu Run.
12 Save the configuration Lsd
  • File/Save
A modified configuration should be saved with a different name from the configuration loaded before the changes since, when running a simulation, the system overwrites the configuration file with the new initial values.
13 Run the newly configured model Lsd
  • Run/Run
It is possible that an attempt to run a modified model does not succeed. The system prevents running model configurations not adapt, issuing messages to fix the errors. For example:
  • model already run. The configuration is derived from the last step of a simulation exercise, and not from a "fresh" configuration file;
  • data non initialized. If the number of Objects is increased, then the newly created instances contain Parameters and lagged Variables that must be initialized before they can be used for a simulation run.
It is also possible some configurations are initialized with non-consistent values, causing, for example, divisions by zero errors. The simulation results are, in these cases, erratic, and may even cause sudden crashes of Lsd. To avoid this keep always a copy of a working configuration, and try few changes per time.
